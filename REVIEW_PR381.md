## Verdict

**Not ready for production.** The architecture (outer-loop continuation wrapping the untouched Q-limit solver, dispatch-clean device abstraction, delta-based stamping) is sound and several subsystems are excellent — but the PR has one merge blocker confirmed by its own CI, four more release blockers that produce silently wrong results in realistic use, a set of mathematical gaps in the continuation for which I have concrete counterexamples, and a performance profile that is currently the *opposite* of the stated goal, with a clear, incremental path to fix it.

## A. Merge blockers

### A1. The tap path cannot run on any released PowerSystems.jl — CI is red and the feature's tests have never executed
`control_metadata.jl:97,106,112,119` call `PSY.get_regulated_bus_number`, `get_tap_limits`, `get_number_of_tap_positions`, `get_voltage_setpoint` on `TapTransformer`. None exist in any released PSY (verified against v5.10.0, v5.11.1, and `main`); they are PSY issue #1684 / open PR #1705, labeled *PowerSystems 6.0*. `Project.toml` still declares `PowerSystems = "^5.10"`.

**CI confirms it:** job `Julia 1 - ubuntu-latest` on the PR head fails with
`UndefVarError: 'get_regulated_bus_number' not defined in 'PowerSystems'` at `control_metadata.jl:97`, and because `runtests.jl` aborts on the first error, **the entire 569-line `test_discrete_control.jl` suite has never run in CI** — every green claim about the feature is untested in the merge pipeline.

*Fix:* gate the merge on the PSY release shipping #1705 + compat bump, or feature-detect (`hasmethod`) with an ext-key fallback so the package loads and tests run on the compat floor.

### A2. Reported branch flows are wrong on exactly the controlled branches
`apply_parameter!` mutates only the square Y-bus (`nzval`). The per-arc matrices `arc_admittance_from_to/to_from` — used by `solve_power_flow!` (solve_ac_power_flow.jl:227–232) to compute every reported `Sft/Stf` and loss — are built once from the *original* taps/angles and never updated. After the controller moves a tap 0.9→1.1, the reported flow on that transformer is computed with the old tap while voltages reflect the new one: up to ~10–20 % flow error, broken nodal balance in results, and for a PAR the immediate contradiction that its own reported flow does not equal the `p_target` it just regulated to. The LCC path already special-cases this ("their self-admittances may change"); controlled taps/PARs need the same treatment. Found independently by three reviewers.

*Fix:* cache arc-admittance nzval offsets per device (mirroring `nz_offsets`) and update them in `apply_parameter!`, or recompute controlled-branch flows from device state as the LCC block does. Add a flow-parity test vs a freshly built system at the snapped parameters.

### A3. Final device settings are reported nowhere; PSS/E export is self-inconsistent
The converged tap ratio / shunt steps / PAR angle / FACTS b live only in `d.current`. Nothing writes them back to the PSY system, the results dataframes, or the exporter — `psse_export.jl:924/890` read `PSY.get_tap`/`get_alpha` from the untouched system. An exported `.raw` therefore pairs continuation-adjusted voltages with pre-continuation device settings: PSS/E will not reproduce the exported solution, and the user cannot learn what the solver chose. For a feature whose purpose is commercial-tool parity, the solved settings are the primary output.

### A4. Multi-period solves silently corrupt device bookkeeping
Shunt/FACTS deltas are written per-time-step (`withdrawals[bus, ts] += d.current − b`) but `d.current` is a single scalar carried across the ts loop, so at ts ≥ 2 the first update is applied against the wrong baseline (offset by `b_final(ts−1) − b_initial`). Taps have the dual problem: the shared Y-bus carries ts = 1's snapped tap into ts = 2's starting network, and after the loop holds the *last* ts's taps for all downstream consumers. No guard rejects `time_steps > 1` with controls, and PSI's PCM loop is exactly this pattern. Found independently by three reviewers.

*Fix (cheapest first):* reject `time_steps > 1` + controls at construction; or reset devices to per-ts baselines at `_control_continuation!` entry; or store per-ts applied state.

### A5. `ControlledTap.alpha` is hardcoded 0.0, but PNM stamps `TapTransformer` with `exp(PSY.get_α·im)`
PNM v0.24.0 (`Ybus.jl`): `tap = get_tap(br) * exp(PSY.get_α(br)*1im)`, and PSY defines `get_α(::TapTransformer) = −(π/6)·winding_group_number` when the vector group is set. For any winding-grouped transformer the delta updates subtract an unrotated "old" term from a rotated slot — every update corrupts the off-diagonals. *Fix:* `alpha = PSY.get_α(tx)` at enrollment + a parity test with a nonzero winding group.

## B. Applied-math assessment of the continuation algorithm

The under-relaxation algebra itself is **correct** (verified by hand and against finite differences: local slope m = 1 + ω(g′−1); max sigmoid slope exactly |hi−lo|·S/4; ω ≤ (1−θ)/(1+|g′|) gives m ≥ θ). But the surrounding claims do not hold, and I have executable counterexamples (scalar replicas of the exact code logic; scripts referenced below):

1. **The convergence guarantee is conditional, and the condition is unstated.** dV/dp is measured *once* at the base point and reused for all 100 passes and all steepness values. (a) If the true sensitivity **flips sign** along the trajectory (classic OLTC reverse action in stressed conditions), the loop becomes positive feedback and walks the parameter *monotonically* to the wrong limit — no sign reversals, so the oscillation guard never fires; simulated: V dragged from 0.995 to 0.974 with `osc = 0`. (b) If the true gain grows > 2× above the probed value the map slope goes negative and healthy devices oscillate into a freeze (simulated at 4×). The doc's "guarantees monotone convergence at every steepness without re-measuring" is unsound as stated. *Fix:* secant-update dV/dp each pass from the already-computed (Δp, ΔV) — zero extra solves — and freeze on detected sign change.

2. **dV/dp = 0 is treated as negative gain and "reliable."** A device whose controlled bus is PV/REF-pinned (or electrically insensitive — remote controlled buses on stiff networks) probes y1−y0 = 0, is *not* frozen, and its target becomes a constant steep sigmoid of the pinned voltage: simulated, a switched shunt at an over-voltage generator bus switches **all** capacitor blocks in (physically backwards), settles at the rail, and the loop reports `regulation_complete = true`. Three reviewers found this independently. *Fix:* treat |dVdp| below a noise floor as unreliable → freeze + warn; skip devices whose controlled bus is PV/REF at base.

3. **The contraction argument is single-device; coupled devices freeze.** For N devices the iteration matrix is I + Ω(G − I) with off-diagonal coupling the probe never measures; the in-phase mode of N same-bus devices has gain ≈ N·g_self. θ = 0.5 covers N = 2 only asymptotically. Simulated (exact Gauss-Seidel replica): 2 parallel identical taps → one frozen at pass ~11 and the steepness ramp permanently stuck; N = 5 → 2 frozen; N = 8 → 4 frozen. Parallel transformer banks regulating one bus are ubiquitous. *Fix:* divide ω by the number of co-located controllers (or probe group gains), and fix the freeze/schedule interaction (next item).

4. **One frozen device permanently blocks the steepness ramp for everyone.** A frozen device returns `Inf` forever ⇒ `settled` is never true again ⇒ S never ramps ⇒ `regulation_complete` is unreachable and the remaining pass budget is burned idling. The oscillation counter is also *cumulative and never reset* (doc says "consecutive"), with no deadband on `sign(p_tgt − p_now)`, so one legitimate overshoot per steepness ramp (7 stages × limit 3) can freeze a healthy device. *Fix:* count frozen devices as settled for scheduling (PSS/E lock-and-continue), reset the counter per stage, ignore sub-tolerance reversals.

5. **The 100-pass global budget is insufficient even for one healthy device.** Because ω ≈ 0.5/(0.05·S·|dVdp|) shrinks like 1/S, each steepness stage needs ~13 passes to re-settle; a single in-band tap (dV/dp = −0.5) needs 88 of the 100 shared passes; dV/dp = −0.3 or a wide-range shunt exhausts the budget and exits with only a `@warn`. *Fix:* per-stage budgets, scale-aware settle tolerance (`CONTROL_PARAM_TOL = 1e-5` is 5e-5 relative for a tap but 1e-6 relative for a 10 p.u. shunt — also fix), and the local (not global-max) sigmoid slope in ω once the gain is re-measured per pass.

6. **`regulation_complete` can be a false positive.** When `_continuation_to!` cannot move at all (first min-step trial fails), it returns `reached == start`; the 0.0 change counts as *settled*, S ramps, and the loop declares success with an unregulated device (simulated: "complete" after 7 passes with V 0.04 p.u. off setpoint). *Fix:* return a `moved::Bool` and treat requested-but-unapplied moves as unsettled/frozen.

7. **Shunt snapping is suboptimal and physically unrealizable.** The greedy-floor + single ±1 refinement misses joint moves (counterexample: blocks [+1.0×1, −0.4×3], target 0.5 → returns 0.0, error 0.5 vs optimal 0.1; brute force over 1,000 random mixed-sign banks: worst extra error 0.57 p.u.). Independently, PSS/E switched-shunt blocks activate *cumulatively in listed order* — the reachable set is a prefix chain — and ~40 % of the greedy outputs are not on that chain, i.e. not achievable by the physical device. *Fix:* snap to the nearest point of the precomputed prefix-sum chain — simultaneously optimal over the realizable set, order-respecting, O(Σsteps), allocation-free.

8. **The PAR reuses voltage-tuned steepness on a power quantity.** S ∈ [100, 5000] is calibrated for O(0.01–0.1) p.u. voltage errors; dP/dα ≈ |yt| ≈ 10 p.u./rad makes gbound O(250–12,500), ω collapses to 2e-3…4e-5, and the sigmoid saturates for any flow error > ~1/S. Simulated: a PAR needing +0.3 rad reaches 0.09 rad in 100 passes (2.1 p.u. residual flow error). *Fix:* non-dimensionalize per device: S_eff = S / (|dydp|·(hi−lo)).

9. **Provenance note.** The Agarwal/Pandey papers embed the sigmoid in the NR Jacobian with λ-continuation. The outer-loop reinterpretation here — and its convergence argument — is **original to this PR**, not paper-backed; the citations cover the sigmoid law and the S schedule constants only. That is fine, but the doc should say so, and the argument needs the repairs above before it earns the "guarantee" language.

## C. State-consistency defects in the engine (major)

1. **Q-limit flips are sticky across rolled-back probes.** `_snapshot_voltage` captures only V/θ. `_check_q_limit_bounds!` one-way flips PV→PQ and clamps Q with no reset anywhere; a probe excursion (δ ≈ 2e-4 tap) that trips a marginal generator permanently changes the problem even after rollback, the final bus-type set becomes path-dependent on the λ trajectory, and a probe measured *across* a flip yields dVdp jumps of O(100) that poison ω. *Fix:* include the `bus_type` and `bus_reactive_power_injections` columns in the snapshot tuple.
2. **Diverged injections drift the problem definition under distributed slack.** Every residual evaluation (including on diverged iterates) writes P/Q back into `data` (`_setpq`); the next residual construction re-derives `P_net_set` from those arrays (ac_power_flow_residual.jl:67). For PQ/ZIP buses the bookkeeping telescopes exactly (verified algebraically) and single-REF self-heals — but with distributed slack the diverged `slack·factor` at PV buses becomes the new setpoint after every failed trial, and backtracking is *designed* to produce failed trials. Final answer solves a drifted problem with `converged = true`. *Fix:* snapshot injections, or pin `P_net_set` once at continuation entry.
3. **The snap-failure safety net starts from the diverged state.** When the post-snap solve fails, `_restore_device_group!` runs immediately: `_restore_one!` snapshots the *diverged* iterate as its "last converged state" and its early-return path solves from it. The pre-snap converged voltages are never restored — the restore path operates from the worst possible start exactly when it is needed. *Fix:* snapshot before snapping; restore on failure before entering the restore loop.
4. **Cross-family name collisions in `snap_and_restore!`.** One `Dict{String,Float64}` spans all four device families; PSY only enforces per-type name uniqueness, so a tap and shunt named alike cross their stashed values and `_restore_one!` then walks a tap toward a susceptance — **unclamped** (line 284 applies raw interpolated p). *Fix:* per-group index-aligned vectors (also removes the allocation); clamp in `_restore_one!`.
5. **LCC state is outside every snapshot.** On LCC systems the residual writes per-ts LCC taps/thyristor angles back into `data.lcc` on every evaluation; a diverged trial leaves corrupted (possibly NaN) LCC state that no rollback restores, so every subsequent retry warm-starts from it and the time step cascades to snap-failure. *Fix:* extend the snapshot to the LCC columns when `n_lcc > 0`, or reject controls+LCC until covered.
6. **FastDecoupled is neither invalidated nor rejected.** `FDCacheKey` uses `objectid(power_network_matrix)`, which in-place nzval mutation cannot change: B′/B″ stay factored at the pre-move network for the entire continuation. Answers stay correct (true residuals) but convergence degrades and the engine misreads it as infeasibility. The `ControlledPhaseShifter` docstring *claims* invalidation happens; no code does it. LM/GD/Homotopy combos are likewise unrejected (functional, extremely costly). *Fix:* mutation counter in the key, or reject FD×branch-controls at construction.

## D. PSS/E fidelity defects in metadata sourcing (major)

All verified against fetched PSY parser sources (`pm_io/psse.jl`, `pti.jl`, `power_models_data.jl`):

1. **Shunt range double-counts BINIT.** For MODSW 0/1/2 the parser stores `Y = BINIT` (total in-service admittance) and *zeroes* `initial_status` ("to avoid double counting"). Building `b_min/b_max = b0 ± Σ blocks` on top of b0=BINIT means a fully-on capacitor bank gets range [0.5, 1.0] instead of [0, 0.5]: it can never switch its capacitors *off* under high voltage and claims phantom headroom. Wrong for essentially every raw-parsed shunt with BINIT ≠ 0.
2. **Transformer ext keys don't match the parser.** The parser writes `CONT1`, `COD1`, `RMI1/RMA1/NTP1` (winding-suffixed) and **no VSET** (the VMI1/VMA1 band is not persisted at all); the code probes `NREG/SWREM/RMIDNT` and unsuffixed `RMI/RMA/NTP/VSET`. Net effect once A1 is fixed: every remotely-regulating tap silently regulates its own to-bus; real ratio-limit/step data is ignored; the "legacy ext scrape" fallback documented in comments is dead code. Also: v35 shunt `NREG` is a *node* (node-breaker), `RMIDNT` is a character device-ID — neither may be treated as a bus number.
3. **PAR setpoint is an initial-condition field that defaults to 0.0.** `p_target = PSY.get_active_power_flow(ps)`; the parser never sets `pf` for transformers, so every raw-parsed COD1=3 PST enrolls targeting **zero MW** with a ±180° default angle band — the controller actively destroys the interchange it should hold. Must come from the PSS/E flow window (needs PSY to persist VMI1/VMA1) or an explicit setpoint; warn+skip otherwise.
4. **`admittance_limits` semantics are a trap.** The midpoint-as-voltage-setpoint works only because the parser stuffs (VSWLO, VSWHI) into a field PSY documents as *admittance* limits. An API-built system following the documented semantics (e.g. (−0.3, 0.3) p.u.) yields vset ≈ 0 and the sigmoid slams the shunt to its inductive rail with no diagnostic. Add a plausibility guard (vset ∈ [0.5, 1.5] p.u., else warn+skip) and pursue the upstream rename.
5. **Deadband and RMPCT semantics dropped.** PSS/E holds V anywhere inside [VSWLO, VSWHI] and shares regulation by RMPCT; this PR pins to the band midpoint with an S=5000 sigmoid (transition ~8e-4 p.u.) and lets every co-located controller drive the full error. Result: over-switching PSS/E would never do, systematic mid-band voltage bias, and device fights on multi-controller buses. Implement the deadband error (`max(0, V−hi) + min(0, V−lo)`) and RMPCT-scaled sharing.
6. **Hard-error posture bricks realistic files.** MODSW ≥ 3 (real in interconnection cases), block-less `SwitchedAdmittance` (the PSY *constructor default* — merely enabling the flag on such a system throws via `_validate_shunt!`), series-FACTS records with SHMX=0, and unresolvable buses each abort ALL of `PowerFlowData` construction. PSS/E warns and locks. Convert per-device errors to warn+skip-as-locked. Relatedly, reduction handling is inconsistent (tap terminals: raw `KeyError`; shunts: silent drop even though their susceptance *is* folded into the parent bus) and ignores `reverse_bus_search_map` that every other init path uses; and `ntp < 2 → 33` silently converts locked changers into active 33-position controllers while making `_validate_tap!`'s check unreachable.

## E. Performance: the honest verdict and the path to the claim

**As merged, this would be dramatically slower than commercial adjustment loops — the opposite of the goal.** Verified from the loop arithmetic (and reproduced numerically):

- `_continuation_to!` starts at `MIN_LAMBDA_STEP = 1e-3` of the interval on *every* call and grows 1.5×: **exactly 16 successful full `_solve_with_q_limits!` calls per device move**, no matter how easy the move (the dead constant `INITIAL_LAMBDA_STEP = 1.0` suggests full-step-first was the intent).
- Probing costs 2 full nonlinear solves per device; 7 steepness stages × ~13 passes each × N devices × 16 sub-solves ⇒ **~23,000–68,000 full inner solves per time step** for a 10k-bus/200-device case (minutes to hours), vs a handful of extra factorizations for PSS/E-class sensitivity stepping (seconds).
- Every inner solve rebuilds the residual (O(nnz) union-find subnetworks), copies the Jacobian structure (MB-scale), and **redoes the KLU symbolic factorization** — the sparsity pattern is provably invariant across the whole continuation (the PR's own structure cache proves it). Rect/mixed additionally rebuild `Y_bus_eff` (full complex copy + ZIP fold) per solve. Note: the `PolarNRCache` described in `.claude/CLAUDE.md` **does not exist in this tree or on main** — there is no value-level reuse to lean on yet (and when it lands, `apply_parameter!` must invalidate its constant-Z snapshots or shunt control silently stops working — leave a landmine comment now).

**Fix path (multiplicative, all incremental):**
1. **Adjoint sensitivities instead of probes:** dy/dp = −uᵀJ⁻¹(∂F/∂p) via one `tsolve!` on the already-factored Jacobian — one linear solve per device (or per pass, enabling the per-pass gain refresh that repairs B1/B3) instead of 2 nonlinear solves. PNM's KLU cache has `tsolve!`; AAFactorCache does not — gate on backend. ~100–1000× on the probe phase.
2. **Full-step-first with halve-on-failure** in `_continuation_to!`: 16 → 1–2 solves per move (~8–16×).
3. **Batch all device updates per pass into one inner solve** with rollback-and-split on failure (~N× on pass cost; the relaxation theory already treats the pass as one damped joint update).
4. **Tolerance laddering** across steepness stages (loose at S=100, full only at S_max) ~1.5–2×.
5. **Persist residual/Jacobian/symbolic factorization across the continuation's inner solves** (invalidate only on bus-type flips) — 2–5× per solve.

Combined, a time step costs *tens* of solves, and the robustness features this PR genuinely adds over commercial tools (guaranteed rollback, oscillation lock, snap-and-restore feasibility recovery) become a real differentiator instead of an apology.

## F. TR/Iwamoto, homotopy, and cache work — mostly excellent, two fixes

**Verified correct (positive results):**
- The ρ rewrite is exactly main's formula with cached squared norms — and it **fixes a genuine main-branch defect**: pred = actual = 0 gave ρ = NaN, all comparisons false, δ never shrunk → stall to maxIterations. The new −Inf path is comparison-safe throughout.
- The generalized Iwamoto quartic is the mathematically right object (exact ‖·‖² of the local quadratic model along the *dogleg* direction, interpolating the true residual at μ=1); the 3-arg wrapper reproduces classical Iwamoto exactly (coefficients are 2× main's old cubic — identical roots). Brute force over 1,008 cases across scales 1e-8…1e6: zero out-of-range returns, worst suboptimality 1.1e-7. Dogleg scratch aliasing audit clean; LCC systems remain square. LM scratch buffers ordering-safe.
- The AC-Jacobian structure cache's core claim is **true**: `_create_jacobian_matrix_structure` pushes all four block entries regardless of bus type, so the pattern is PV↔PQ-flip invariant; template values are safe today; the homotopy `pq_diag_nz` freeze is unreachable (Hessian rebuilt per solve call); the CHOLMOD `issuccess` rewrite also *fixes a latent crash* (zero-pivot LDL′ previously threw uncaught `ZeroPivotException`).

**Two fixes needed:**
1. **`issuccess` does not detect indefiniteness under simplicial LDL′** (empirically demonstrated on Julia 1.12.6 with the exact `fixed_structure_CHOLMOD` call sequence: indefinite 2×2 and 1600-dim matrices report `issuccess = true`, `is_ll = 0`): the τ-regularization loop can exit with an indefinite Hessian, breaking the homotopy line-search descent premise. One-line fix, verified: wrap the factorization in `CHOLMOD.@cholmod_param final_ll = true` (what stdlib `cholesky!` does); do **not** use `isposdef(F)` (MethodErrors on 1.12.6). Not a regression — the old solve-probe was equally blind — but the new comment claims a PD check that isn't one.
2. **Structure-cache hardening:** the key omits REF/subnetwork placement of the distributed-slack structural columns (add a sorted-REF fingerprint — low reachability, cheap insurance), and the comment "*Every nzval entry is rewritten by J(time_step)*" is false — many entries (REF rows, PV Q-slots, zero-participation cross terms) rely on pristine template zeros, which is safe only under the fresh-copy-per-solve + no-PQ→PV-restore invariants; state those invariants instead.
